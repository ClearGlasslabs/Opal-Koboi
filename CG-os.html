Data and Ontology

The ontology is the contract between humans, agents, data, and policy.

Every mission object carries confidence, classification, compartments, release markings, valid time, system time, and lineage. Agents do not query raw tables directly; they receive policy-filtered object sets and execute governed ontology actions that create drafts until humans or policy gates promote them.

Core entities

  • Mission, IntelEvent, Entity, Relationship, Alert, Case, ActionPackage, AgentRun, FeedbackSignal, EvalCase, ChangeProposal.
  • Entity types: person, organization, asset, account, device, location, infrastructure, facility, vehicle, document.

Relationships

  • ObservedIn, AssociatedWith, DerivedFrom, SupportsMission, ContainsAlert, ProducedByAgent, EvaluatedByFeedback.
  • Edges include evidence IDs, assertion source, approval status, temporal validity, and confidence.

Permissions

  • Need-to-know controls at row, column, object, entity, edge, tool, and action levels.
  • Coalition access requires release authority plus matching release markings; role membership alone is insufficient.
AI and Agent Design

Copilots advise. Agents prepare. Humans authorize operational significance.

Analyst Copilot

Explains alerts, summarizes timelines, drafts RFIs, identifies evidence gaps, and emits citation-bound notes.

Commander Copilot

Produces mission posture briefs, risk projections, courses of action, resource impact estimates, and confidence caveats.

Multi-agent workflow

TriageAgent scores incoming events; EnrichmentAgent resolves entities; CorrelationAgent links historical patterns; SummaryAgent drafts intel; RecommendationAgent prepares action packages for approval.

Tool-using agents

Tools query ontology object sets, retrieve governed evidence, generate intel products, open draft cases, and prepare action packages. Consequential actions require approval tokens.

Self-Improvement Loop

CG-OS gets better without changing goals by itself.

1

Capture signals

Operator corrections, query logs, accepted and rejected recommendations, alert dispositions, mission outcomes, latency, precision, recall, citation coverage, and trust ratings become FeedbackSignal objects.

2

Generate evals

Feedback is transformed into held-out eval cases, regression suites, drift tests, prompt-injection tests, policy tests, and mission-context benchmarks.

3

Propose upgrades

AIP Eval Steward agents produce ChangeProposal records for prompt diffs, workflow edits, model-route changes, and heuristic thresholds with expected lift, risk score, blast radius, and rollback target.

4

Approve and deploy

Human reviewers approve changes; Apollo canaries signed artifacts; monitors compare A/B cohorts; rollback triggers on safety, access, accuracy, latency, or operator-trust regression.

from pydantic import BaseModel, Field
from enum import Enum
from datetime import datetime

class Severity(str, Enum):
    info="INFO"; low="LOW"; medium="MEDIUM"; high="HIGH"; critical="CRITICAL"

class IntelEvent(BaseModel):
    event_id: str
    mission_id: str
    event_type: str
    event_time: datetime
    classification: str
    compartments: list[str] = Field(default_factory=list)
    release_marks: list[str] = Field(default_factory=list)
    confidence: float
    payload_hash: str
    lineage_refs: list[str]

class PolicyContext(BaseModel):
    actor_id: str
    roles: list[str]
    compartments: list[str]
    coalition: str | None = None
    purpose: str

def can_read_event(ctx: PolicyContext, event: IntelEvent) -> bool:
    has_compartment = set(event.compartments).issubset(set(ctx.compartments))
    is_mission_role = "analyst" in ctx.roles or "commander" in ctx.roles
    return is_mission_role and has_compartment and ctx.purpose in {"investigation", "command"}

class ArtemisAgentTools:
    def __init__(self, ontology, policy, audit):
        self.ontology = ontology
        self.policy = policy
        self.audit = audit

    async def query_related_entities(self, ctx, entity_id: str, depth: int = 2):
        decision = await self.policy.allow(ctx, "ontology.relationship.read", {"entity_id": entity_id})
        await self.audit.record(ctx.actor_id, "query_related_entities", entity_id, decision)
        if not decision.allowed:
            raise PermissionError(decision.reason)
        return await self.ontology.graph_neighbors(entity_id=entity_id, depth=min(depth, 3), policy_ctx=ctx)

    async def prepare_action_package(self, ctx, alert_id: str, recommendation: dict):
        decision = await self.policy.allow(ctx, "action_package.draft", {"alert_id": alert_id})
        await self.audit.record(ctx.actor_id, "prepare_action_package", alert_id, decision)
        if not decision.allowed:
            raise PermissionError(decision.reason)
        return await self.ontology.create_draft("ActionPackage", {
            "alert_id": alert_id,
            "recommendation": recommendation,
            "status": "AWAITING_HUMAN_APPROVAL",
            "created_by": "AIP.RecommendationAgent",
        })

async def self_improvement_job(feedback_store, eval_runner, proposal_store):
    signals = await feedback_store.load_recent(min_count=250)
    eval_suite = build_eval_suite(signals, holdout_ratio=0.35)
    candidates = propose_prompt_and_route_changes(signals)

    for candidate in candidates:
        report = await eval_runner.run(
            suite=eval_suite,
            artifact=candidate.artifact,
            gates={"precision_min": .92, "recall_min": .86, "p95_latency_ms_max": 1800, "unsafe_outputs": 0},
        )
        if report.passed and report.delta.operator_trust >= 0.03:
            await proposal_store.create({
                "change_type": candidate.kind,
                "proposed_diff_uri": candidate.diff_uri,
                "eval_report_uri": report.uri,
                "risk_score": report.risk_score,
                "approval_status": "IN_REVIEW",
                "rollback_target": candidate.current_version,
            })
Security and Governance

Zero-trust execution with immutable evidence.

Access control

AuthN uses enterprise SSO, hardware-backed MFA, mTLS, short-lived service tokens, device posture, and session risk. AuthZ combines RBAC, ABAC, relationship-based mission assignments, and policy-as-code.

Compartmentalization

Entity-level and relationship-level controls prevent spillover. Coalition views are generated as derived object sets with explicit release markings and source redaction.

Model governance

Prompts, tools, model routes, evals, policies, and workflow graphs are versioned artifacts. No model can silently change its own objectives, authorities, tools, or deployment cohort.

Scenario Walkthrough

A live event becomes a safer, smarter future workflow.

At 02:14 UTC, a live intel feed emits a high-noise infrastructure anomaly. Foundry validates schema, deduplicates payloads, applies classification labels, and materializes an IntelEvent. AIP TriageAgent receives only the mission-scoped object set it is authorized to see, assigns preliminary severity, and asks EnrichmentAgent for related entities. CorrelationAgent finds a historical pattern but marks confidence as provisional because one source is stale. SummaryAgent drafts a cited brief in Gotham. RecommendationAgent prepares an action package, but CG-OS blocks execution until a commander approves. The operator rejects one entity link, downgrades severity, and adds a correction. That correction becomes an eval case, lowers trust in the stale source, adjusts a heuristic proposal, and triggers a prompt diff that tells future agents to penalize the same lineage pattern. Reviewers approve the diff after the eval harness shows higher precision with no recall loss. Apollo deploys it to a 5% canary, monitors latency and false positives, then promotes or rolls back based on gates.
